# steering単位のコミット & プッシュ

引数で渡された情報から `.steering/` 配下の対象ディレクトリを特定し、
**ユーザーの承認を得てから**、その steering に対応する変更を
ステージング → コミット → プッシュまで実行します。

引数: steering を特定できる情報（ディレクトリ名の一部・機能名・日付など）

## 実行方法

```bash
claude
> /commit-steering sw-navigation
> /commit-steering レベルC ナビゲーション
> /commit-steering 20260618-sw-navigation-levelc
```

## 手順

### ステップ1: steering の特定

1. `.steering/` 配下のディレクトリ一覧を取得する（`ls .steering/`）。
2. 引数の文字列を手がかりに、一致する steering ディレクトリを探す。
   - ディレクトリ名（`[YYYYMMDD]-[機能名]`）に対する部分一致を最優先する。
   - ディレクトリ名で一意に決まらない場合は、各 steering の
     `requirements.md` の `## 概要` を読み、内容との一致で絞り込む。
3. 結果に応じて分岐する:
   - **1件に特定できた** → ステップ2へ。
   - **複数候補がある** → 候補を一覧で提示し、`AskUserQuestion` で
     ユーザーにどれか選んでもらう。選択後ステップ2へ。
   - **0件** → 候補が見つからない旨と `.steering/` の一覧を提示し、終了する。
     （勝手に推測してコミットしない）

### ステップ2: 対象ファイルとコミットメッセージの算出

以下を**自動で**組み立てる（ファイル選択・メッセージともユーザーに確認しない）:

1. **対象ファイルの自動算出**
   - 特定した steering の `requirements.md` / `design.md` / `tasklist.md` を読み、
     本文中で言及されている実装ファイルのパスを抽出する。
   - `git status --porcelain` で変更中のファイル一覧を取得する。
   - 「steering が言及するパス」∩「実際に変更されているファイル」を
     ステージング対象として確定する。
   - 加えて、その steering 自身のディレクトリ
     （`.steering/[特定したディレクトリ]/` 配下、主に `tasklist.md`）の
     変更もステージング対象に含める。
   - **注意**: `.steering/` には複数の作業が並行して存在しうる。
     `git add -A` や `git add .` は使わず、必ず算出した対象ファイルのみを対象にする。
2. **コミットメッセージの自動生成**
   - `requirements.md` の `## 概要` と `design.md` の内容を理解し、
     その steering で何を実装したかを表す簡潔な日本語1行メッセージを生成する。
   - リポジトリ既存の慣習（体言止め「〜の実装」等。例:「バックグラウンド処理の実装」）に合わせる。
   - **このメッセージはユーザーに確認しない。** 自分で適切に決定する。

### ステップ3: steering 承認（必須ゲート）

`AskUserQuestion` で、**特定した steering で合っているか**だけを確認する:

- 特定した steering ディレクトリ名（と、参考としてステージ予定ファイル一覧を併記）
- 「この steering で commit & push を進めてよいか？」を Yes/No で問う

**この承認を得るまで `git add` / `git commit` / `git push` を実行してはならない。**
承認するのは steering の特定が正しいかどうかのみ。
ファイル選択とコミットメッセージは自動確定済みであり、改めて確認しない。
（特定が誤っている場合はユーザーが別の steering を指定し直す）

### ステップ4: ステージング → コミット → プッシュ

steering の承認後、算出したファイルのみを対象に一気通貫で実行する:

```bash
git add <算出したファイル ...>
git commit -m "<自動生成したメッセージ>"
git push
```

- `git rev-parse --abbrev-ref HEAD` で**現在のローカルブランチ**を確認し、
  そのブランチのまま同名リモートへ push する（`main` 直 push もそのまま行う）。
- upstream 未設定エラーが出た場合は `git push -u origin <現在のブランチ>` で push する。
- いずれかのコマンドが失敗したら、出力をそのまま提示して停止する
  （勝手に `--force` 等で回避しない）。

### ステップ5: 結果報告

```markdown
✅ steering「[ディレクトリ名]」をコミット & プッシュしました

- コミット: [短縮ハッシュ] [メッセージ]
- ステージしたファイル: N件
- push 先: origin/[ブランチ]
```

## 注意事項

- ステアリングの特定ミスは手戻りが大きいため、**steering の承認なしに git 操作を行わない**こと。
- 確認するのは「steering の特定が正しいか」のみ。ファイル選択とコミットメッセージは
  自動で確定し、ユーザーには確認しない。
- ステージングは steering に関連するファイルのみに限定し、
  並行作業中の他 steering の変更を巻き込まない。
- コミットメッセージは steering の内容を理解した上で、
  リポジトリ既存の簡潔な日本語スタイル（体言止め「〜の実装」等）に合わせて自分で決める。
